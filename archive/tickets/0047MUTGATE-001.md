# 0047MUTGATE-001: Reconcile cargo-mutants timeouts (exit 3) in the in-diff gate instead of hard-failing

**Status**: COMPLETED
**Priority**: MEDIUM
**Effort**: Small
**Engine Changes**: Yes — `.github/workflows/ci.yml` (`mutants-in-diff` job); no crate, event, schema, fixture, or doc-semantics changes.
**Deps**: None. Builds on the inline fix that added `--timeout 183` to the in-diff `cargo mutants` invocation (commit on branch `implemented-0046tuisimpla`).

## Problem

The `mutants-in-diff` job (`.github/workflows/ci.yml`) runs `cargo mutants --in-diff` and
then post-processes its exit status. The guard whitelists only exit `0` (all caught) and
exit `2` (FOUND_PROBLEMS / misses) before reaching the baseline-miss reconciliation
(`comm -23` against `.cargo/mutants-baseline-misses.txt`). cargo-mutants exits **3 on
TIMEOUT**, which falls through to the `"tool/run failure"` branch and hard-fails the job
**before any baseline reconciliation runs**.

With the auto-set ~20s test-timeout (now pinned to 183s by the inline fix), a timeout only
occurs when a mutation genuinely hangs (e.g. an infinite loop introduced on a guarded line)
or when the workspace test suite legitimately exceeds the pinned ceiling. In either case a
timed-out mutant is a *survivor* (uncaught), which the standing perimeter already has a
sanctioned channel for: the accepted-baseline-miss file. The scheduled shard jobs reconcile
timeouts through `tools/merge-mutation-shards.py`; the in-diff job has no equivalent path, so
an *accepted-baseline* timeout still hard-fails the PR gate. This is an inconsistency, not a
correctness hole — the inline `--timeout 183` fix makes the gate deterministically green for
the current tree — but it leaves the gate fragile to any future genuinely-slow or
hanging-but-accepted mutant on a guarded line.

## Architecture Check

1. Treating exit `3` as miss-bearing and routing `mutants.out/timeout.txt` entries through the
   same normalized `comm -23` reconciliation as `missed.txt` is cleaner than the alternatives:
   (a) widening the whitelist to accept `3` unconditionally would let a genuine new hanging
   survivor pass silently — a fail-open regression; (b) raising `--timeout` further only defers
   the cliff. Reconciling timeouts the same way misses are reconciled makes the in-diff gate's
   survivor handling consistent with the scheduled shard path and fails closed on any timeout
   not present in the accepted baseline.
2. No backwards-compatibility shim or alias path is introduced; the change edits the existing
   guard in place.

## What to Change

### 1. `.github/workflows/ci.yml` — `mutants-in-diff` job guard

Change the exit-status handling so exit `3` (TIMEOUT) is treated as miss-bearing rather than a
hard tool failure: keep exit `1` (and other non-{0,2,3} codes) as genuine tool/run failures,
but on exit `3` proceed to artifact + reconciliation handling. Fold `mutants.out/timeout.txt`
(when present) into the survivor set that is normalized (drop line:col) and compared with
`comm -23` against `.cargo/mutants-baseline-misses.txt`, exactly as `missed.txt` is today. Any
timed-out mutant not in the accepted baseline must fail the gate; an accepted one passes.

## Outcome

Completed: 2026-06-22

Changed `.github/workflows/ci.yml` in the `mutants-in-diff` job so
cargo-mutants exit `3` is treated as survivor-bearing output rather than a
tool/run failure. The job now folds `mutants.out/timeout.txt` together with
`mutants.out/missed.txt`, normalizes both through the existing line/column
stripper, and compares the combined survivor set against
`.cargo/mutants-baseline-misses.txt` with `comm -23`. Any new timed-out mutant
still fails closed; an accepted-baseline timeout can pass through the same
channel as an accepted miss.

Added CI guard coverage in
`crates/tracewake-core/tests/anti_regression_guards.rs` and
`crates/tracewake-core/tests/ci_workflow_guards.rs` so status `3`,
`mutants.out/timeout.txt`, and the combined `new_survivors` reconciliation
remain locked.
