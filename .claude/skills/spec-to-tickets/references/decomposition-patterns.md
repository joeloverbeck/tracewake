# Decomposition Patterns (Step 3)

Analyze the spec and identify discrete work units:

- Each ticket is a **reviewable diff** — small enough for comfortable manual review.
- Map **dependencies** between tickets and name them in each ticket's `Deps`.
- Determine **priority ordering** from the dependency graph and criticality. **When the spec supplies an explicit closure / implementation / remediation order** (e.g. a §Closure-order or §Implementation-posture sequence), treat that order as the dependency-ordering backbone and the default ticket sequence — it encodes the author's intended landing sequence (and often the only safe order for interlocked atomic cutovers, where a later step won't compile until an earlier one lands); diverge only as a flagged decomposition decision (§Sizing rules, placement/ordering directives).
- Ensure **every spec deliverable is covered** — no silent skipping. If a deliverable seems wrong, flag it with the 1-problem / 3-options / 1-recommendation format rather than omitting it.
- **Either-or deliverables** — distinct from a *wrong* deliverable: when a deliverable enumerates ≥2 doctrine-permissible implementation options and the spec leaves the choice open, do not silently pick one. Surface the choice as a Step-4 Question with a recommendation and bake the selected option into the ticket. If the user delegates the decision ("you decide" / "align with the docs"), resolve it by reasoning against the authority tiers (foundation → architecture → execution → reference), cite the deciding clause in the ticket's Problem/Architecture Check, and treat the spec's own listed-first option as a tiebreak only when the tiers are silent — mirroring `reassess-spec`'s delegated-resolution rule. Approval that is silent on the Question follows the Step 4 rule: the recommended default applies and is announced inline. Under the Step-4 auto-mode carve-out (no approval round at all), the same resolution applies: a Question with a presented recommendation resolves to it, announced in the table Notes for post-hoc redirect; a Question without a clean recommendation holds the wait-gate like an Issue (HARD-GATE clause (c)). **Spec-assigned implementer choices are not open choices**: when the spec itself assigns the decision to the implementer as a recorded choice (a "the decision is the implementer's, recorded in <artifact>" directive — the established pattern in this repo's hardening lineage), no Step-4 Question fires; the ticket instead carries the choice-and-record obligation — options named, recording destination named — in its Assumption Reassessment / What to Change, and the Step-4 table Notes may mark it `implementer-recorded choice` for visibility.
- Use the spec's **Non-goals** / **Out of scope** / **Forbidden changes** sections to populate each ticket's Out of Scope — these are pre-validated non-goals.

## Defect-class closure (quantified and conditional deliverables)

The intent-degradation failure this section prevents is documented lineage history, not hypothesis: spec 0019's required corrections said "derive both censuses structurally — every apply arm…", prescribed a freshness/supersession rule fix, and said "add `sleep.rs` + `work.rs` (and `eat.rs` if nontrivial)"; the tickets landed one census derived (the other hand-extended), the rule fixed on one of its two consuming surfaces, and the conditional clause silently dropped — all caught one full audit pass later as `ORD-HARD-053`–`055` (spec 0020). Decomposition is the last cheap place to stop this: quantifiers, rule-classes, and conditionals must be expanded against the codebase at ticket-writing time, never forwarded as prose for the implementer to interpret.

- **Quantified deliverables enumerate their members.** When a deliverable or required-correction states its surface as a quantifier or class — "every apply arm that writes X", "both censuses", "all record kinds", "each terminal family", or a named defect pattern — expand the quantifier against the actual codebase (grep/read during Step 2/3) into an explicit member list, bake the list into the owning ticket's Acceptance Criteria (one criterion per member, or a member table), and show the expansion count in the Step 4 table Notes (`every apply arm → 6 arms enumerated`). A ticket whose acceptance criteria restate the quantifier un-expanded can be satisfied by fixing only the spec's cited instance.
- **Rule fixes enumerate sibling surfaces.** When a correction fixes a *rule* on one named surface (a freshness/supersession rule, a validation gate, a classifier), grep for the rule's other implementations and consumers — the Assumption-Reassessment item-7 blast-radius pattern applied to fixes, not only renames/removals. Every sibling surface either joins the ticket's scope, gets its own ticket, or is recorded as an explicit justified exclusion (Step 4 Notes + the ticket's Out of Scope). "The spec only cited surface A" is not an exclusion rationale: the spec cited where the defect was *found*, not the defect class's extent.
- **Conditional clauses resolve at decomposition time.** A clause of the form "(and X if nontrivial)", "if cheap", "where applicable", or "if any remain" is a decomposition-time determination, not implementer discretion. Resolve each with a recorded yes/no and a one-line rationale — in the Step 4 table Notes and the owning ticket's Assumption Reassessment — and on a *yes*, fold X into the enumerated scope. An unresolved conditional is a silently-skipped deliverable, exactly the failure the Ticket-fidelity guardrail forbids.
- **Step 6 closes the loop.** The coverage mapping confirms every quantified deliverable's member list appears in some ticket's Acceptance Criteria (or carries recorded exclusions) and every conditional clause has a recorded resolution — see SKILL.md Step 6 item 3.

## Deliverable-coverage categories

Most deliverables get their own ticket. These categories are exempt from per-deliverable ticket production — document them in the Step 4 summary and Step 6 coverage mapping instead:

- **No-change deliverable** — the spec explicitly states no change is needed ("No new crate"). Note it if non-obvious; produce no ticket.
- **Cross-spec consumer-side deliverable** — the work is consumed by a sibling phase's tickets rather than producing a discrete output here. The change is needed, but it happens in another spec's decomposition. Note the ownership as `cross-spec: <sibling-phase>` in Step 4 and `not ticketed (cross-spec: …)` in Step 6.
- **Distributed deliverable** — one conceptual deliverable encoded as a contract distributed across multiple tickets (e.g. an applicability rule carried as a predicate on each affected module, with no single owner). Note it as `distributed across <ticket-list>`; produce no separate "container" ticket. When the deliverable is a *named subsystem* enumerating concrete file/symbol changes (not just one applicability predicate), it is only safely distributed if **every** named surface lands in some ticket's Files to Touch — list each constituent surface and confirm its owner; a surface owned by no ticket is an uncovered deliverable masquerading as distributed, not a true distributed deliverable. Distinguish a definition-layer change from a selection-layer change: a spec requiring both an autonomous-*selection* change (the chooser) and an action-*definition* change (the action's own payload/validation) names two surfaces, and a ticket owning only the selection surface leaves the definition surface uncovered.
- **Informational-pointer deliverable** — a section whose body only points to sibling deliverables ("handled in §X above and §Y below"). The actual work is in the pointed-to siblings, each ticketed. Note the routing; produce no ticket.
- **Already-landed deliverable** — the intent was already satisfied before decomposition: either an edit landed during a prior in-session skill invocation (e.g. `/reassess-spec`'s user-authorized in-session edit), OR a pre-existing commit already shipped it (commonly the spec-package commit that bundled the spec's own documentation/status deliverables — ledger, errata, an architecture clarification doc). Confirm the landing (`git log -- <path>`, or read the artifact) rather than assuming. Produce no ticket; record `not ticketed (already landed during <skill> session <date>)` or `not ticketed (already landed in <commit>)` in Step 6.
- **Inferred deliverable** — named or implied by §Purpose / §Acceptance-evidence / §Exit-criteria but absent from §Deliverables. This DOES get a ticket; document the inference in the ticket's Assumption Reassessment with a citation to the motivating section, and flag it in Step 4 Notes as `inferred from <section>`. When the motivating reference is ambiguous or would expand scope, route through a Step 2 Issue instead.

## Sizing rules

- **Merge** — when multiple deliverables share the same file set and cannot be implemented independently, merge into one ticket. Note merged deliverables in Step 4 Notes.
- **Split** — when a single deliverable is too large for one reviewable diff (it spans, say, a contract surface + a schema + a validation rule + an event emitter), split by logical sub-concern (e.g. write path vs. read/verify path). Note the split in Step 4 and show `D<n>→00x+00y` in Step 6.
- **Atomic-cutover exception to Split** — some specs forbid incremental landing: an anti-incremental / anti-contamination thesis where any merge that lands a piece while leaving the old path live (behind a wrapper, a fallback, or a still-present bypass) would violate the spec's integrity (the spec typically says so — "remove it, don't wrap it"; "do not keep a partial convenience layer"). For such a deliverable, a deliberately-large single "atomic flip" ticket that performs the whole cutover in one reviewable diff is correct and overrides the Split rule. Build the new path additively in prior tickets (kept unwired), then flip + delete-all-bypasses in the one atomic ticket, and land the regression lock immediately after. Record the override rationale in the ticket's Architecture Check and flag the deliberate size in the Step 4 table Notes. This is distinct from a normal Large deliverable that simply *could* be split — here splitting is unsafe, not merely inconvenient.
- **Local compile-atomicity (schema-field retype)** — a second, narrower atomicity case independent of any spec thesis: a single-crate schema-field *retype* (e.g. `field: Option<String>` → a typed struct, or a public authority flag → a sealed capability type) whose consumers all live in the same workspace is a *local* atomic unit. Every consumer must change in the same diff or the tree won't compile, so the field retype plus its in-workspace consumer updates stay in one ticket even when sizeable — splitting it would land an uncompilable intermediate state. This is a compilation constraint, not a spec directive; size the ticket to the retype + its consumers and flag the deliberate size in the Step 4 table Notes. A reseal/retype that turns a public field or flag into a private/sealed surface is neither a pure schema *extension* nor a pure *rename/removal*, so its ticket carries BOTH `_TEMPLATE.md` Assumption-Reassessment item 6 (the visibility/shape change — name the consumers and whether it is additive-vs-breaking) AND item 7 (the removal blast radius of the old public surface), keyed as distinct numbered items. Because such reseals routinely live in files the item-6 verification net's filename list does not name (`runtime/receipt.rs`, `runtime/command.rs`, `runtime/session.rs`, `debug_capability.rs` — not `*schema*`/`state.rs`/`view_models`/`projections`), SKILL.md Step 5 and Step 6 treat any reseal carrying item 7 as an item-6 trigger regardless of filename, so the additive-vs-breaking analysis is still verified when the seal lands in a runtime-authority module.
- **Spec-enumerated granularity is a floor, not a 1:1 mapping** — when the spec's deliverables / required-areas / work-items enumerate the work at a stated granularity, treat that as the author's intended *granularity floor* — the coarsest acceptable grouping (equivalently, a *minimum ticket count*: "floor" bounds the count from below, so split finer freely but merge below the spec's item count only via the Merge exception) — not a binding one-ticket-per-item rule. Honor it as the default, but apply the Split rule when a labeled item exceeds one reviewable diff (the highest-risk surface — kernel growth, a full new module/crate — usually does). The **Merge** rule below is the sanctioned exception to this floor — deliverables sharing a file set and not independently implementable may collapse below the spec's item count. Surface every divergence from the spec's item count in the Step 4 table (`D<n>→00x+00y` for a split; note the merged deliverable set for a merge), and offer the spec-granularity alternative ticket count, direction-aware: when the decomposition went finer than the spec, offer the coarser (spec) count; when it merged coarser than the spec, offer the finer (spec) count — so the user can opt to the spec's granularity either way. Under the Step-4 auto-mode carve-out the offer collapses into the table note — flag the split/merge and state the spec-granularity count there, and the user redirects post-hoc.
- **Placement/ordering directives that would produce dishonest intermediate states** — when the spec's own prose assigns a deliverable's placement or landing order (e.g. "documentation corrections land with group 1") and following it literally would create a dishonest intermediate state (a conformance/doc row describing an enforcement that has not landed yet, an evidence claim ahead of its proof), the decomposition may diverge as a **flagged decomposition decision**: place the deliverable with the ticket that makes its claim true, surface the deviation in the Step 4 table for approval (it rides the normal approval/carve-out flow), and record it in the Step 6 summary. This is not a spec defect — no `/reassess-spec` round-trip is required — the spec's *intent* (honest evidence) takes precedence over its placement letter.
- **Same file / same new directory** — when all deliverables modify the same file or land under one new directory (a new crate, a new module dir), decompose by logical section/module, not by file boundary. Each ticket targeting a different section is a valid reviewable diff. Note the shared file once in Step 4.
- **Intra-batch create-then-modify chains** — when ticket B's Files to Touch contains a `(modify)` entry for a file ticket A creates `(new)` in the same batch, declare `Deps: A` on B explicitly. The file-creation is a load-bearing structural dependency (B fails at implementation time if A hasn't landed). This is distinct from §Shared-file overlaps (Step 6 item 5), which covers parallel adds by mutually-independent tickets to a shared file — whether pre-existing or created in-batch by a common foundation ticket they each `Deps:` on. Annotate the `(modify)` entry with its in-batch creator (e.g. `(modify — file created by <PREFIX>-A)`) so a reviewer reading B in isolation — and the Step 5 `test -f` existence check — can tell the file is created in-batch rather than missing.
- **Ticket-creating deliverables** — when a deliverable's output is itself a ticket file (a spec mandates "record the deferral as a real ticket in `tickets/`"), reserve the next free number in the batch's namespace at decomposition time (e.g. a 14-ticket batch reserves `-015` for the authored deferral ticket), record the reservation in the Step 4 table Notes and the Step 6 summary so a later batch in the same namespace cannot collide, and declare the create-then-reference chain explicitly: the authoring ticket creates the file; any sibling whose commands or acceptance criteria check that file declares `Deps` reaching the author. When the authored-ticket *count* is implementation-discovered (a triage or audit whose output volume is unknowable at decomposition time), reserve a starting number plus an open range instead (`-NNN onward`, count implementation-discovered), record the open reservation the same way, and note that the next batch in this namespace must check the actual high-water mark before claiming numbers.
- **Greenfield foundation batch** — when a spec introduces the first code for a new module or crate (no pre-existing code tree for that surface — common when a later phase opens a new area, and how all of Phase 1 began), every ticket's Files to Touch for that surface is legitimately all-`(new)`; do not treat that as a suspicious under-statement. Confirm the surface is genuinely new (`find crates -name '*.rs'`, grep the crate tree) rather than assuming the whole repo is greenfield — completed phases already have a landed workspace. The structural ordering is carried by intra-batch create-then-modify chains (above), and an unbuilt deliverable's consumers are the sequenced later phases, not present code — mirroring the unbuilt-surface note in `references/codebase-validation.md` and in `reassess-spec`'s `references/codebase-validation.md`. One file is a near-universal shared-file overlap in any module-registering workspace: the crate-root module-declaration hub. Distinguish a **new crate** (a new workspace member — register in BOTH the workspace members list `Cargo.toml` AND the new crate's root `lib.rs`) from a **new module inside an already-landed crate** (register in the parent `lib.rs` / `mod.rs` ONLY — `Cargo.toml` is untouched, since no workspace member is added). Most foundation tickets that *open a new crate* `(modify)` `Cargo.toml`; a keystone that adds a module to an existing crate (e.g. a new `runtime/` module in an already-landed `tracewake-core`) registers in `lib.rs` alone and must not carry a spurious `Cargo.toml` entry. Flag this hub explicitly at Step 4 — in the table Notes or as a dedicated hub line in the same approval message, not only at Step 6 item 5 — since it is a guaranteed N-way mechanical merge surface among otherwise-parallel siblings. A **test-side greenfield** variant is common (a new capability registry, conformance runner, or other integration-test support-module set): there the surface is all-`(new)` under `crates/<crate>/tests/`, the create-then-modify hub is the shared `tests/<name>/mod.rs` plus its consuming `tests/*.rs` target — **not** `lib.rs`/`Cargo.toml`, and no workspace-members edit is involved — and that `tests/` mod hub is flagged the same way.
- **Shared-file merge hubs (any batch, not only greenfield)** — the same N-way merge risk applies in a *landed* workspace whenever ≥3 mutually-independent siblings each `(modify)` one pre-existing file — a central test file (e.g. `anti_regression_guards.rs`), a shared registry, a `mod.rs`. This is the common shape in a hardening / anti-contamination batch, where many tickets each append a distinct test fn to one existing suite. Count the distinct tickets touching each shared file and flag any file touched by ≥3 otherwise-parallel siblings at Step 4 (table Notes or a dedicated hub line in the same approval message) — not only at Step 6 item 5 — so the user sees the coordination cost before approving. The greenfield registration hub above is one instance of this general rule. **Linear-chain caveat**: when the siblings sharing a file are *sequentially dependent* (a `Deps` chain — e.g. a behavioral-remediation batch where `002→003→…` each `(modify)` one central `scheduler.rs`), the file is a **sequential-edit coordination surface**, not a parallel-merge hub: the edits stack in chain order and do not merge-conflict, so flag it for landing-order awareness rather than as an N-way mechanical merge (and say so when surfacing it, so the user does not over-read the coordination cost). The ≥3-mutually-independent-siblings hub remains the parallel case that genuinely conflicts.

## Ticket shapes

**`Engine Changes` field value by ticket kind** (the field is `<None|Yes — list …>` per `tickets/_TEMPLATE.md`): set `Yes` for tickets adding source code, **tests** (test-additions count — e.g. §Mutation-remediation kill tickets, negative-fixture tickets), or **CI config** (`ci.yml`, `.cargo/mutants.toml`); set `None` for **doc-only** (doctrine strengthening, conformance-truthing), **evidence-only** (a standing mutation run+record), and **acceptance-only capstone** tickets. A mixed hardening batch carries all of these in one decomposition (e.g. `0053`: a test-only manifest ticket, doc-only doctrine/truthing tickets, a CI-config governance ticket, an evidence-only mutation-run ticket, an acceptance-only capstone) — set each ticket's field by its own kind rather than defaulting the whole batch one way.

**Trailing-shape co-occurrence**: a single spec closeout deliverable often bundles docs-finalization **and** exit-evidence; split it into both shapes below — a cross-cutting docs ticket (Deps = the surface-citing implementation tickets) plus a capstone (Deps = the leaf set, including the docs ticket). The spec closeout artifact — (a) a `docs/4-specs/SPEC_LEDGER.md` entry/row added post-landing (the usual case for a forward implementation spec, whose own Status need not flip), and/or (b) a spec Status → `Done` flip (for a retroactive spec) — defaults to the **capstone**, since it is gated on exit evidence passing; the docs ticket carries doc content and any earlier status update, not the completion artifact — unless the spec's §Documentation-updates assigns the ledger entry to the docs ticket, in which case the §Cross-cutting docs ticket `Deps: <capstone>` exception applies.

**Staged spec (not yet at its `docs/4-specs/` home)**: when the source spec still lives under `specs/` with a declared final home of `docs/4-specs/NNNN_…` (not yet promoted), the `docs/4-specs/SPEC_LEDGER.md` entry and the final-home move are deferred to spec promotion — outside the ticket batch. Record them in the Step 6 cross-spec follow-ups rather than ticketing them, unless the spec's own §Documentation-updates assigns the ledger entry to a docs ticket.

**Hardening / certification specs** (the namespace section's non-phase class — e.g. the `0009`–`0014` series) — and, on the same axis, **feature/capability specs in the parallel `specs/NNNN` series** (e.g. `0047`, a post-`FIRST-PROOF-CERT` capability spec): both are the common variant of the above, with a different destination: their accepted final home is `archive/specs/`, **not** `docs/4-specs/` (they are staged in `specs/`, then archived on acceptance, never promoted to live `docs/4-specs/`), and their `SPEC_LEDGER.md` entry is the *Archived implementation specs* row. The same deferral applies — defer the ledger row and the `archive/specs/` move to spec acceptance/archival (per `docs/archival-workflow.md`), recorded as a Step 6 cross-spec follow-up, not ticketed.

**Behavioral-remediation hardening spec** — when a hardening spec's deliverables are *engine-code rewrites* against a numbered finding-set (the common `NNNN_*_HARDENING_SPEC` genre — e.g. `0048` — that fixes conformance defects in `crates/`, distinct from the §Audit/certification-evidence shape, which only *runs* tests and records witnesses, and the §Mutation-remediation shape, which kills mutation survivors). Decomposition shape: when several findings interlock through a shared authority surface (a field reseal, a transaction/prepare-commit boundary, a caller refactor that will not compile until the new path exists), build the new path **additively** in early tickets, kept unwired (the §Sizing rules atomic-cutover + local-compile-atomicity pattern) — an additive early ticket may land as a `(modify)` adding new private code *inside an existing file* (e.g. discovery/registry helpers added to a large `scheduler.rs`), not only as a `(new)` module; note it as a merge-hub contributor (§Shared-file merge hubs) and leave module extraction as an implementer-recorded choice when the spec assigns the structure; perform the bypass-deletion + caller-flip in one **atomic-cutover** ticket — never a split that lands a piece while leaving the old path live behind a wrapper; then layer the evidence tickets (held-equal differential, replay/corruption suite, parity); when the spec mandates a mutation campaign (a §6.3-style focused-plus-standing run), land it as one standalone `Engine Changes: None`, evidence-only run+record ticket (`Deps` the evidence ticket), modeled on the §Mutation-remediation §Run + triage ticket but **without** a `.cargo/mutants.toml` change when the spec says the perimeter already covers the seams — its recorded denominators/artifacts feed the capstone. **But when the finding-set *names specific survivors to close with new witnesses*** (not merely a fresh run of an already-witnessed perimeter — e.g. `0051` §4.8 requires adding production-path `insert_observation` / `embodied_subject_key` witnesses before the run), that mutation ticket is instead `Engine Changes: Yes` (witness test-additions + the run), keyed like a §Mutation-remediation kill ticket with the per-survivor witness members enumerated in Acceptance Criteria (§Defect-class closure); reserve the `Engine Changes: None` run+record form for when the witnesses already exist and only a fresh campaign is owed. The mutation ticket owns exactly the survivors *not* already closed by a finding ticket: if that residual set is empty it stays the `Engine Changes: None` run+record form, otherwise it is `Engine Changes: Yes` owning only the residual witnesses + the run (e.g. `0052`'s mutation ticket owned just the unowned `StuckDiagnostic::deserialize_canonical` survivors after the per-finding tickets closed the rest), with those residual members enumerated in Acceptance Criteria per §Defect-class closure. When the spec also mandates an *enforced standing barrier* — a required public-boundary conformance CI lane and/or branch-protection governance, distinct from the mutation run — split that into its own `Engine Changes: Yes` CI-config ticket (`Deps` the corpus + run tickets; modify `ci.yml`, not `.cargo/mutants.toml` when the perimeter already covers the seams), and record the required-check name + branch-protection confirmation as operational evidence in the capstone artifact (a workflow file alone cannot prove branch protection). Close with an acceptance-only capstone (§Capstone integration ticket). Sequence the whole batch on the spec's §Closure-order when it supplies one (SKILL.md Step 3). Each ticket carries the usual AR items 5/6/7 keyed distinctly per the reseal/schema/removal rules. The `SPEC_LEDGER.md` row + `archive/specs/` move defer to spec acceptance (§Hardening / certification specs above), recorded as a Step 6 cross-spec follow-up. Exemplar to calibrate against: `0048FOUCONHAR-*` (frontier reseal + atomic one-tick transaction + replay temporal projection + holder-known interval flip + evidence + capstone) once it lands.

**Doc-doctrine / doctrine-alignment amendment batch** — when the spec is a doctrine-tier amendment *proposal* whose deliverables are edits to doctrine docs (`docs/{0-foundation,1-architecture,2-execution,3-reference}/NN_*.md`) rather than code — the recurring tracewake foundation→architecture→execution cascade, e.g. the `0026`→`0027`→`0028` lineage — every ticket's Files to Touch is all-`(modify)` markdown doc paths, and the code-centric Step 5/6 machinery is uniformly N/A: there are no Rust symbols to spot-grep, the item-6 schema-shape heuristic does not fire (no `envelope.rs`/`*schema*`/`view_models`/`state.rs` modify-targets), and Test-Plan verification is grep-based landing + boundary checks plus an invariants-alignment review, not `cargo` tests — state each N/A explicitly in the ticket (the `None — documentation-only` Test-Plan form) rather than omitting it. Each ticket carries an owner-approval execution precondition per the target tier's amendment authority (constitutional sign-off for foundation; ordinary owner approval for architecture/execution/reference) and must not be applied by convention. Producer/consumer `Deps` apply to a doctrine *standard* one ticket introduces in a shared doc (e.g. a general rule that sibling tickets' specific requirements reference), not only to code symbols; the shared doctrine doc is the merge hub to flag (§Shared-file merge hubs). Exemplar to calibrate against: the `archive/tickets/0027ARCTIEDOC-*` series. **Decomposition axis**: when this batch's deliverables map ~1:1 to docs (each deliverable has one primary home doc — the `0027`/`0032` shape), either axis works and per-doc is conventional. When the deliverables span docs *many-to-many* (each deliverable touching several docs, and each doc receiving several deliverables — the entangled case, e.g. an execution-tier amendment whose every proof obligation also adds a slice to the diagnostics/observability doc), decompose **per-target-doc, not per-deliverable**: per-doc preserves 1:1 doc↔ticket ownership and yields zero merge hubs, whereas per-deliverable makes any shared *consolidation* doc (a diagnostics / observability / index home that nearly every deliverable touches) an N-way merge hub (§Shared-file merge hubs). Per-doc decomposition makes a multi-doc deliverable *distributed* (§Deliverable-coverage-categories), so verify coverage **per doc-surface** — every doc a deliverable names lands in some ticket — not only per deliverable. **AR item 5 for doctrine tickets**: a doc-amendment ticket that *governs* a fail-closed-validation / actor-knowledge / deterministic-replay surface by doctrine — without touching code — still carries `_TEMPLATE.md` Assumption-Reassessment item 5 on the §Substrate-only-tickets basis (SKILL.md Step 5): name the governed enforcement surface and confirm the doctrine change introduces no leakage/nondeterminism path. This holds even though the item-6 schema-shape heuristic does not fire for doc batches. **Doctrine amendments as a precursor sub-batch within a *code* spec**: the common tracewake feature-spec shape routes its doctrine amendments (a §5 "substance + home" section) as a single "establish approved semantics before code" deliverable preceding the code work (e.g. `0047` §4.9 item 1). Decompose that one deliverable by the *same* per-tier rule used here — split by amendment authority (foundation constitutional sign-off vs ordinary owner approval for architecture/execution/reference), one ticket per tier (or per-doc when a tier's docs are many-to-many) — and make the resulting doctrine tickets `Deps` of the first code ticket, since the semantics must land first. **Operationalizing-doctrine exception (doctrine-with/after-code)**: when the doctrine *operationalizes* an executable mechanism the code introduces — a fail-closed acceptance manifest, a wording guard, a computed-pass rule — rather than establishing semantics the code then implements, the dependency direction **inverts**: the doctrine ticket `Deps` the mechanism ticket and lands with or after it, because a doctrine ticket documenting a mechanism that does not yet exist is dishonest. The spec's own wording is the signal — "lands together with the … code," "so the rule is executable, not merely written" (e.g. `0053` §6.1, whose §6.1 doctrine ticket `Deps` the F5-06 manifest ticket rather than vice versa). Distinguish "approved-semantics-before-code" (doctrine-first, the default above) from "operationalize-an-executable-rule" (doctrine-with/after-code). Each such doctrine ticket is a doc-only ticket carrying the all-`(modify)` markdown / `None — documentation-only` Test-Plan / AR-item-5-on-the-substrate-basis treatment above, even though the surrounding spec is a code spec. Surface the doctrine-deliverable's finer-than-spec split (1 deliverable → N tier tickets) at Step 4 per the granularity-floor rule, offering the collapse-to-one alternative.

**Audit / certification-evidence spec** — when the spec is a *certification audit* whose deliverables are evidence-collection obligations producing a single acceptance artifact (the `P0-CERT` class — and the `SPINE-CERT`/`EPI-CERT`/`ORD-LIFE-CERT`/`FIRST-PROOF-CERT` audits sequenced after it), not code diffs. Distinct from the §Hardening/certification-specs note above (which addresses only the archival destination) and from the doc-doctrine batch (which edits doctrine prose): a per-gate audit ticket *runs* existing tests/fixtures and records witnesses against current code — though audit specs of this lineage routinely also bundle non-per-gate sub-tickets that edit config or add test-only harness code (below). **The `§`-numbers in the components below are EPI-CERT-specific exemplar anchors — map each to the *target* spec's actual section layout, since sibling audit specs reuse the same numbers for different content (e.g. ORD-LIFE-CERT §6.1 is the pass-condition mapping, not EPI-CERT's compile-fail corpus; its relational/generated evidence lives in §6.4; and it has no compile-fail corpus at all, so the Cross-cutting negative-evidence ticket below is N/A there).** Decomposition shape:

- **Surface the structural choice at Step 2/4** — a certification-audit spec is not always ticketized the same way and the spec rarely dictates it: per-gate evidence ticket + capstone (faithful to a §7-style per-gate proof list), grouped-by-evidence-machinery (fewer merge hubs), or do-not-ticketize (run the audit as one investigation and ticketize only the remediation it surfaces — what such specs typically anticipate). The spec having no explicit anti-decomposition directive (Step 1) does **not** make per-gate the default; present the options as a Step-2 Issue / Step-4 Question with a recommendation. When the recommended option is per-gate, present its full Step-4 decomposition table alongside the structural Question — the table is that option's concretization, so selecting per-gate approves the table in one round-trip; only a non-recommended selection forces a re-table. Present the structural choice and its table in **one assistant message** so per-gate selection approves both at once: emit the options as plain-text `(a)/(b)/(c)` bullets with the table inline, OR — if `AskUserQuestion` is used for the discrete choice — place the full table as text in the *same* message as the tool call (text renders before the tool UI). A standalone `AskUserQuestion` that defers the table to a later message forces the two-round-trip (answer the choice, then separately approve the table) that the one-round-trip rule exists to avoid.
- **Per-gate evidence ticket** — `Engine Changes: None`; Files to Touch is the acceptance artifact only (the documentation/evidence-only shape); the Test Plan runs the gate's existing tests/fixtures and the deliverable is the captured behavior witnesses, provenance rows, and typed diagnostics (use the `None — documentation/evidence-only` Test-Plan form). The single acceptance artifact is a **create-then-modify N-way hub** — one scaffold ticket creates it `(new)` at `reports/<NNNN>_<slug>_acceptance.md` (the repo convention — e.g. `reports/0042_ord_life_cert_..._acceptance.md`; archived to `archive/reports/` on spec acceptance per `docs/archival-workflow.md`), every gate ticket `(modify)`s its own section and `Deps` the scaffold — so flag it per §Shared-file merge hubs, noting the "merge" is low-risk append-only on distinct sections; recommend sequential landing. When the spec's *first* audit point is itself the unified-baseline / coherent-artifact-set proof (the `P0-CERT` / `FIRST-PROOF-CERT` shape — gate-01 *is* the scaffold/baseline work), merge it into the scaffold ticket rather than giving it its own ticket, and offer the un-merged count per the direction-aware granularity-floor rule (§Sizing rules). This differs from the `ORD-LIFE-CERT` shape, where gate-01 is a behavioral point kept separate from the scaffold (exemplar `archive/tickets/0042ORDLIFCER-001`/`-002`).
- **Mutation-perimeter ticket** — when the audit spec *itself* makes mutation a certifying requirement (the `EPI-CERT` §7 / `SPINE-CERT` shape), one `Engine Changes: Yes` ticket *modifies* the standing `.cargo/mutants.toml` (and the CI mutation job, only if it pins a narrower file list — resolve this conditional at Step 2/3 by grepping the CI mutation job per §Defect-class closure, so the Step-4 Files-to-Touch lists or excludes `ci.yml` concretely at approval rather than carrying an unresolved conditional) to the configured union, runs the standing-continuity + expansion census, and *creates* the survivor triage register. Its Files to Touch are the config + register + its artifact section — **not** the report alone. The glob/file expansion is a quantifier to enumerate per §Defect-class closure; reuse the predecessor register shape. Survivor remediation is still routed out (this ticket records the posture, it does not fix).
- **Cross-cutting negative-evidence ticket** — when the spec mandates a consolidated compile-fail / boundary-corpus matrix cited by several gates (the `EPI-CERT` §6.1 shape), author it once as a single `Engine Changes: None` ticket (`Deps` the scaffold) that the per-gate tickets reference, rather than duplicating the matrix across them. Its corpus member list is a quantifier to enumerate in Acceptance Criteria.
- **Relational / metamorphic capstone-harness ticket** — when the spec requires a paired-run / property harness proving a relational (hyperproperty) statement the single-run gates cannot (the `EPI-CERT` §EPI-11 / §6.2 shape), it is `Engine Changes: Yes — test-harness only`: it *adds* assertions to existing certification suites, so its Files to Touch is those suites **plus** its artifact section. Distinct from the acceptance-only verdict capstone (which adds no logic).
- **AR items 4/5 on the evidence-consumer basis** — the ticket *audits/reads* an actor-knowledge / replay / fail-closed-validation enforcement surface rather than modifying it, so item 5 still applies (name the surface; confirm the evidence collection introduces no leakage/nondeterminism path and that the artifact's debug rows stay observer-only); item 4 restates the gate's motivating `INV-NNN`. The item-6 schema-shape heuristic does not fire for a per-gate evidence ticket (its only modify-target is the report — but the mutation-perimeter and harness sub-tickets above do modify config/test files, so apply the heuristic to them normally).
- **Remediation is routed out, not pre-authored** — any gate failure is recorded `fail` with responsible layer; the fix becomes a separate `<gate> scoped remediation` ticket/spec, since what needs fixing is unknowable until the audit runs. Each gate ticket's Out of Scope states this.
- **Capstone** — the verdict/consolidation ticket is an acceptance-only capstone (no new production logic — see below); it `Deps` the gate leaf set and renders the single verdict.
- Exemplars to calibrate the whole shape against: `archive/tickets/0038SPICEREVE-*` (SPINE-CERT — scaffold + per-gate + mutation + capstone) and `archive/tickets/0036P0CERPOS0008-*` (P0-CERT).

**Mutation-remediation spec** — when the spec's thesis is *fixing* a failed gate's mutation posture (the `0037`/`0039` lineage: a `<gate>-CERT mutation remediation and replacement certification` spec), not auditing it. Distinct from the §Audit/certification-evidence spec above, whose *per-gate* tickets only *run* tests and record witnesses: a remediation spec's deliverables are real code diffs — new behavior-witness tests, perimeter/CI config, and conditional production corrections. (Survivor identities are cargo-mutants `path:line:column` entries whose line/column anchors are unreliable — reconcile each by **enclosing symbol + mutation operator**, grep the symbol at the Step 5 spot-grep, and treat the cited line as advisory; this is the recurring deep-researched-spec shape, where symbols are accurate but `:line:col` is not.) Decomposition shape:

- **Perimeter foundation ticket** — one ticket converting the standing mutation config (`.cargo/mutants.toml`), the CI mutation jobs, and the CI guard test to the full standing seam perimeter (`Engine Changes: Yes`; the final campaign run depends on it).
- **Per-survivor-file kill tickets** — one ticket per spec-enumerated survivor **file** by default (the `0039` exemplar's shape). When the spec enumerates survivor *families* finer than files (a §per-survivor-group list grouping by behavior/function *within* a file), that finer list is the granularity *floor*, not a 1:1 mandate: keep small files whole, and apply the **Split rule** (§Sizing rules) when one file's survivor count exceeds a single reviewable diff — split that file by behavior-family/function (so e.g. a 21-survivor `proposition.rs` becomes several tickets while a 2-survivor file stays one), and offer both counts at Step 4 per the direction-aware granularity-floor rule. Each kill ticket is `Engine Changes: Yes`, test-additions-by-default with a conditional `` `<path>` (modify — only if a survivor reveals a real defect) `` production-file entry, the file's survivor count + a member-by-member kill matrix in Acceptance Criteria (per §Defect-class closure — the per-file count is a quantifier to expand), and `cargo mutants -f <file>` as the per-ticket verification boundary. These are mutually independent (`Deps: None` — disjoint source files) but commonly converge on a few central test suites (flag per §Shared-file merge hubs).
- **Run + triage ticket** — one `Engine Changes: None` evidence ticket running the full standing campaign to completion, reconciling every historical survivor plus every newly-discovered one, and producing the triage register (`Deps` the foundation + all kill tickets). Run-discovered survivors needing a fix route to a reserved open range (§Ticket-creating deliverables).
- **Capstone** — an acceptance-only re-proof + replacement-artifact ticket (folding the spec's multi-seam re-proof into one capstone, per §Capstone integration ticket; `Deps` the run+triage ticket) that renders the gate verdict and supersedes the prior artifact. The `SPEC_LEDGER.md` row + `archive/specs/` move defer to spec acceptance (§Hardening / certification specs), recorded as a Step 6 cross-spec follow-up.
- **Completion-dominant variant (no known survivors / lane cannot complete in one process — the `0042→0043`, `0044→0045` shape)** — when the spec's blocking floor is *campaign completion* rather than survivor-killing (the spec says the configured lane timed out or did not complete and records no actionable miss), the shape inverts: the single **Perimeter foundation ticket** splits into completion-infrastructure tickets — trigger/guard convergence, supervisor-retention + completion diagnostic, and, when the lane must be **sharded** to complete, a **set-union merger tool ticket** (a new tool computing the canonical set `C`, each shard's assigned set `Sₖ`, and outcome set `Oₖ`, proving pairwise-disjoint exact union, shipped with its own synthetic-negative suite: missing-shard, duplicate-identity, mismatched-commit/config, truncated-output, overlapping-shard) plus a **sharded scheduled/manual CI-matrix ticket** (`fail-fast: false`, unique `always()` artifact upload, explicit per-shard budgets, a reconciliation job invoking the merger). Up-front per-survivor-file kill tickets collapse to **zero** — all survivor work routes to the reserved `NNN+` open range (§Ticket-creating deliverables), the run+triage ticket `Deps` the infrastructure chain rather than a kill-ticket set, and the capstone's "complete final rerun" precondition re-gates any range ticket that lands.
- **AR items 4/5 on the substrate basis** — each kill ticket restates the survivor file's motivating `INV-NNN` (item 4) and, when it touches a fail-closed-validation / replay / actor-knowledge surface, carries item 5 — and item 6 keyed separately when the kill *establishes a new serialized consumer* (e.g. extends a `projection` checksum input or a `view-model` schema so a survivor accessor becomes consequential): name the schema, its consumers, and additive-vs-breaking. The run + capstone tickets carry item 5 on the evidence-consumer basis (they audit/re-prove those enforcement surfaces). Exemplars to calibrate against: `archive/tickets/0037P0CERTMUTREM-001..003` (minimal, single known survivor), `archive/tickets/0039SPICERMUT-*` (kill-heavy), and `archive/tickets/0043ORDLIFCER-*` (supervisor-completion diagnosis + run-discovered survivors); `archive/tickets/0045FIRPROCER-*` is the completion-dominant / sharded-merger variant once it lands.

### Capstone integration ticket

A single trailing ticket whose scope IS the spec's §Exit-criteria / §Acceptance-evidence, exercising every prior implementation ticket end-to-end. It introduces no new production logic; it exercises the pipeline the earlier tickets composed.

**Deliverable-doubles-as-capstone**: the "no new production logic" constraint applies to *acceptance-only* capstones. When the phase's own verification harness is itself a deliverable (e.g. a golden-trace / deterministic-replay smoke ticket that adds harness code or a new CLI tool), that ticket MAY serve as the capstone even though it ships new test infrastructure — note the dual role in the Step 4 table. And when no single ticket exercises the whole phase end-to-end, acceptance MAY be distributed across the trailing tickets (e.g. no-human-run smoke + deterministic-replay/hash check + no-leak negative test + docs/status-flip); record the distribution in the Step 6 deliverable-coverage mapping instead of forcing a synthetic capstone.

- Its acceptance criteria enumerate the spec's exit-criteria / acceptance-evidence bullets as test sub-cases, mapped to the relevant Phase acceptance gates (`docs/2-execution/03_PHASE_LADDER_GATE_ORDER_AND_CERTIFICATION_SEQUENCE.md`; `docs/0-foundation/12_FIRST_PLAYABLE_SCOPE_AND_ACCEPTANCE_GATES.md`).
- Re-enumerate expected counts from a fixture at test start rather than hardcoding (hardcoded counts go stale).
- **Files to Touch** — `_TEMPLATE.md` makes the section mandatory, but a capstone introduces no production logic, so it is either the new golden-trace / replay-check / smoke test the ticket adds, or `None — verification-only` when the phase is exercised purely by running existing tools (`simulate`, `replay-check`) plus a manual runbook, or — when the capstone assembles a written acceptance/evidence artifact (the §7 acceptance-artifact shape) — that artifact at `reports/<NNNN>_<slug>_acceptance.md` (repo convention; archived to `archive/reports/` on spec acceptance per `docs/archival-workflow.md`). Do NOT list the upstream tickets' files — the capstone exercises them, it does not modify them.
- **Manual-runbook variant** — when an exit-criterion cannot run in the project's test infra — either it requires invoking an LLM-driven skill (not runnable from test code), **or** it is a TUI / manual playability smoke with no automation harness yet — structure the ticket as a hybrid: a *manual runbook* (in the What to Change section) the implementer follows by hand — for an LLM skill: fixture copy strategy, skill invocations against the copy, expected post-skill state with verification commands; for a TUI smoke: the key-path/possession-view walkthrough with each step's expected observable result — plus an automated-test subsection for any test-runnable portion (golden-trace assertions, deterministic-replay/hash checks, no-leak / actor-knowledge negative tests, why-not checks). The acceptance criteria distinguish CI-runnable assertions from implementer-checklist runbook steps.
- **`Deps`** — prefer a single `Deps: <transitive-head-ticket>` when the upstream DAG is a linear chain (the head's own `Deps` reach every prior ticket). When the upstream DAG has parallel branches (e.g. a docs ticket running parallel to implementation tickets), enumerate the **leaf set** — every leaf whose transitive `Deps` collectively cover the full chain.

### Cross-cutting docs ticket

A single trailing ticket whose Files to Touch are markdown docs (READMEs, doc-pack files, skill reference files, `docs/4-specs/SPEC_LEDGER.md` index updates — see the §Ticket shapes `Done`-flip default) that must land atomically once all upstream implementation tickets ship.

- One Files-to-Touch entry per docs surface the deliverable names.
- Acceptance criteria as grep-proofs against the post-implementation tree (exact-string matches for new/renamed symbols, count assertions for status-line updates).
- Out of Scope names the implementation surfaces this ticket does NOT touch, so reviewers know docs-only is the contract.
- **`Deps`** — list every implementation ticket whose surface the docs reference (docs reference each surface independently, so transitive-head is insufficient here). **Exception** — when the docs ticket only annotates *aggregate completion* (a `docs/4-specs/SPEC_LEDGER.md` status flip / "verified by Phase N" note) rather than citing individual symbols, and a capstone ticket already gates every referenced surface, `Deps: <capstone>` is sufficient and more legible than re-enumerating the surface set. Enumerate per-surface only when the docs cite individual symbols/APIs that could go stale independently.
- **When NOT to use** — if each docs change can co-locate with its implementing ticket without a staleness window, per-ticket docs are cleaner. Use this shape when ≥1 docs surface requires all upstream surfaces to exist coherently before it can land (status-line counts, cross-tool tables, multi-surface skill guidance).
