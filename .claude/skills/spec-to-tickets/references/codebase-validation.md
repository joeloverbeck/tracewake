# Codebase Validation (Step 2)

Before decomposing, validate the spec's assumptions against the actual Tracewake codebase and doctrine. Collect everything; surface Issues to the user before Step 3.

**Parallel grep-batch guard**: when validating via `grep` / `find` / `test` calls batched in one parallel turn, guard each with `|| true`. A non-zero exit from one call (grep finding nothing, `test` on a missing path) cancels its siblings. Zero-match results are expected and valid during validation.

**Greenfield / paper-spec specs (the current default for Tracewake)**: the repo has no Rust code tree yet — every spec is a paper plan for future implementation, so the consumer/usage greps have nothing in *code* to grep. This is the normal case here, not the exception. Handle it explicitly rather than skipping silently: a deliverable's consumers are the **planned** later phases named in `docs/2-execution/02_PHASE_LADDER_AND_ACCEPTANCE_GATES.md`, the per-phase docs, the `docs/4-specs/SPEC_LEDGER.md` "next allowed spec", and the spec's own Relationship-to-prior-spec / Phase-entry sections — not present call sites. Record the planned consumers for the audit trail; do not flag a zero-consumer Issue for a paper spec whose consumers are unbuilt-but-sequenced future phases. Once a workspace lands, the consumer/usage bullets grep the crate/module tree normally.

## Full Validation Path (default)

- **File paths** — Glob/Grep every path the spec references and confirm it exists where stated. Targets: spec files (`specs/*`, `docs/4-specs/*`), skill files (`.claude/skills/*/SKILL.md` and `references/*`), doc-pack files (the tiered `docs/{0-foundation,1-architecture,2-execution,3-reference}/NN_*.md` layout), and — once it lands — the code tree. Distinguish existing paths (must exist now) from proposed paths (exist after implementation); proposed paths must not collide with an existing file and must follow conventions (`NNNN_TITLE.md` for specs, `<PREFIX>-NNN.md` for tickets, kebab-case skill dirs under `.claude/skills/`, the tiered numbered doc layout per `docs/README.md` and the per-tier `00_*_INDEX.md`).
  - **Doc-tier path drift**: when a spec cites a doctrine file, verify both the tier folder and the `NN_` prefix — a doc reorg renumbers or moves files, and a stale `docs/0-foundation/02_FOUNDATIONAL_INVARIANTS.md` (the constitution is `02_CONSTITUTIONAL_INVARIANTS.md`) or a wrong-tier citation passes a loose existence check silently.
- **Types / schema / contract fields** — Grep each type, record, struct, or schema/contract field the spec names. The schema/contract-authoritative sources are the architecture tier — `docs/1-architecture/03_EVENT_LOG_REPLAY_PROJECTIONS_AND_SAVE_PACKAGES.md` (event log, envelope, causal graph, snapshots, save package, schema migration, random streams), `docs/1-architecture/04_STATE_CONTENT_AUTHORING_DOMAIN_PACKS_AND_SCHEMA_VALIDATION.md` (entity/component state model, content packs, schemas, fixtures, content validation), `docs/1-architecture/05_CLAIMS_BELIEFS_TRACES_MEMORY_AND_INFORMATION_FLOW.md` (typed claims, belief, memory, actor-knowledge filtering) — and the execution schema/fixture contract `docs/2-execution/09_DATA_AUTHORING_VALIDATION_AND_GOLDEN_FIXTURES.md`, plus Spec 0001's fixture/ontology contracts. Verify field names against these and the actual crate code (once it lands), not against constitutional-invariant prose (the invariants carry *intent*; the architecture/execution docs and code carry the *actual* names, and the two can drift). **Apply this with equal rigor to fields named in the spec's proposed corrections / fix prose, not only fields it names as already-existing** — a field a spec proposes to *write* still needs its host structure's schema confirmed to define it; a wrong field in a proposed fix propagates through decomposition into a ticket exactly as a wrong field in a bug claim does.
- **Functions / commands / surfaces** — Grep each; confirm signature, location, layer placement (kernel vs content vs TUI/view-model vs debug, per `docs/1-architecture/01_SYSTEM_AUTHORITY_RUST_WORKSPACE_AND_BOUNDARIES.md`), and that proposed commands/tools/flags (`simulate`, `replay-check`, …) match existing conventions and the spec's recommended workspace shape.
- **Constitutional-invariant references** — Grep every `INV-NNN` invariant, acceptance gate, or rule the spec cites; confirm the heading still exists in `docs/0-foundation/02_CONSTITUTIONAL_INVARIANTS.md` with the claimed semantics.
- **Spec dependencies** — confirm each predecessor/successor phase or spec in the spec's `authority order` and Relationship-to-prior-spec / Sequencing sections resolves to a file under `specs/` or `docs/4-specs/` (or is an expected not-yet-specced phase per the `docs/4-specs/SPEC_LEDGER.md` / `docs/4-specs/README.md` index). An entry resolving to no file that the spec treats as existing is a broken-dependency Issue; a not-yet-specced sibling the spec correctly marks as such is a LOW annotate-the-reference finding. For a predecessor spec the target claims is accepted/landed, verify the ledger status actually reads that.
- **Enum / consumer blast radius** — when the spec introduces, retires, or changes a string-enum value (an event kind, claim family, visibility scope, view-model field, dispatch token) or a new field on an existing schema/contract, pick the analogous existing sibling and grep every consumer site across `docs/*`, `specs/*`, `docs/4-specs/*`, `.claude/skills/*`, and (once it lands) the code tree. New values need an arm at each dispatch/projection site; retired values need every consumer updated or retired with them. Under-enumeration of the consumer set is the most common way a deliverable silently understates a ticket's true Files to Touch — surface the consumer count explicitly. (Until a code tree exists, the consumers are doctrine/spec/fixture references, not call sites — see the greenfield note above.)

**Flag** any stale reference, missing file, renamed entity, or doctrine-violating assumption as an **Issue**, with one carve-out: **mechanical drift** (a path-convention swap where intent is unambiguous, a capitalization variant, an extension normalization, an unambiguous relocation of the same artifact) propagates the corrected reference silently into tickets — note the propagation inline in the Step 2 summary rather than opening an Issue cycle. **Substantive drift** (renamed type, removed file, shifted semantics, doctrine-violating assumption, a field that doesn't exist on the named structure) is an Issue requiring disposition. Decision rule: if a competent reader would immediately see the spec's reference and the codebase's actual reference are the same artifact (no semantics or scope change), it's mechanical; if the divergence raises a question about intent (rename vs. removal vs. typo vs. API shift), it's substantive.

Present Issues to the user before Step 3. For each, obtain one of five dispositions:
- **fix-before-decomposition** — a small spec correction lands first (via `/reassess-spec`), then decompose against the corrected spec.
- **defer-to-follow-up-ticket** — with a named dependency.
- **reject-with-rationale** — spec too materially wrong to decompose this round; route back to `/reassess-spec`.
- **expand-scope-in-place** — validation surfaced materially understated scope; the spec's intent is preserved but tickets are sized against the wider surface the codebase requires (the spec is not edited, the decomposition is).
- **drop-as-moot** — the deliverable's named target doesn't exist AND its intent is satisfied by sibling deliverables OR is a structural no-op under the actual codebase shape; drop it from the decomposition with a Step 2 note naming the missing target and the covering siblings.

**Brainstorm-produced specs**: when `<spec_path>` was produced by the `brainstorm` skill earlier this session, the Full Validation Path still applies — brainstorm's triage validation is less systematic than `/reassess-spec`'s discipline. Treat brainstorm's in-context exploration as already-gathered evidence for the bullets it demonstrably covered, but run fresh greps for everything it did not explicitly verify (notably the schema/contract-field validation, since brainstorm's spec-drafting does not guarantee proposed-fix fields were schema-checked against `docs/1-architecture/04_*` / `docs/2-execution/09_*`). A multi-item brainstorm triage also writes a companion `docs/triage/<date>-<topic>-triage.md`; check its follow-up-items section and surface those follow-ups in the Step 6 cross-spec follow-ups list.

## Abbreviated Spot-Check Path (when `/reassess-spec` ran in-session)

If `/reassess-spec` was run on this spec in the current session and all findings were resolved, Step 2 may be abbreviated to a targeted spot-check (3–5 greps). Verify at least:

- **(a) Primary references** — the spec's primary doc-pack / spec / type references still exist at the stated paths.
- **(b) Schema version** — if the spec modifies a structured schema/contract (event-envelope fields, entity/component record, public/private view shape, content-manifest entry, fixture contract), verify it hasn't drifted since reassessment.
- **(c) Sibling specs** — no new specs under `specs/` or `docs/4-specs/` reference the same surfaces.
- **(d) Additive extension** — for specs extending an existing schema/contract, consumers have been updated, or the extension is additive-only (new optional field with a default), per the event-schema-evolution rule (INV-020; `docs/1-architecture/03_*`).
- **(e) Rename / removal blast radius** — for tickets renaming or removing a module, doc-governed contract, acceptance invariant, schema field, skill, or exported symbol, grep repo-wide (`docs/`, `specs/`, `docs/4-specs/`, `.claude/skills/`, and — once it lands — the code tree) for every symbol. Any match outside the ticket's Files to Touch either (i) joins Files to Touch, or (ii) splits into a follow-up ticket with an explicit dependency, or (iii) when the match lands in a *sibling spec's* committed deliverables/exit-criteria, is surfaced as a flagged Step 4 consideration (held even under auto mode, since altering a sibling spec's tested behavior is a cross-spec scope decision).
- **(f) New-field / restructured-field spot-check** — when the spec adds a field to, or restructures the shape of, an existing schema/contract (string → object, scalar → enum, single → array, or a `required` → optional relaxation that breaks consumers asserting presence), spot-check that the host structure's architecture/execution docs, validation rules, and producing/consuming surfaces are covered by the spec's deliverables. Route any gap per (e).

After spot-checks, render the exercised sub-checks as a compact inline list (e.g. `Spot-checks: (a) ✓, (b) ✓, (c) skipped — no new sibling specs, (d) ✓, (e) N/A — no renames, (f) N/A — no new field`).

If `/reassess-spec` ran but some findings were **deferred** by the user, treat deferred items as out of scope for decomposition. Note them in the Step 6 summary as "deferred reassessment findings that may warrant separate tickets." Do not silently incorporate deferred findings into ticket scope.

## Validation Rules This Skill Upholds

- **Fail-closed validation (foundation 02/12; arch 13; exec 09)** — when a deliverable introduces validation, verify it stays deterministic and blocking, distinguishes warnings from blockers, rejects unknown fields by default, refuses behavior-looking fields (no-scripting doctrine; content is possibility, not script), and names what failing means (block? warn?). Unaddressed second-order effects are Improvement findings at minimum.
- **Kernel authority / determinism (INV-008, INV-042, INV-017, INV-018, INV-019)** — for deliverables touching legality, state transitions, event application, scoring, RNG, replay, hash, serialization order, snapshots, or traces, verify behavior stays in the authoritative kernel (TUI/view-models presentation-only, the LLM out of authority), no nondeterministic input enters canonical forms, and identical inputs + versions produce identical output. A change to replay/hash/snapshot *semantics* additionally requires a foundational-doc amendment (`docs/4-specs/0001_FOUNDATIONAL_DOC_AMENDMENTS.md`). Violations are CRITICAL/HIGH.
- **Kernel boundary (arch 01)** — for deliverables touching the kernel, verify no domain/mechanic noun leaks into it and the doctrine-mandated dependency direction is preserved. A domain-specific type in the kernel, or an authority inversion, is a boundary-failure CRITICAL Issue.
- **No-scripting / content-is-possibility (foundation 09; arch 04)** — content/fixtures stay typed possibility, parameters, metadata, or traces only; behavior-looking fields and authored outcome chains / director logic are a CRITICAL Issue.
- **No-leak / no telepathy (INV-024, INV-006; arch 05/10)** — for deliverables affecting view-model projection, possession, previews, traces, debug views, notebooks, or replay exports, verify no path lets hidden information reach a viewer the actor-knowledge filter forbids. Missing firewall is CRITICAL.
- **LLM/language boundary (INV-042; foundation 11; arch 06)** — speech-act/text deliverables must keep the LLM behind validated extraction and out of world authority. An LLM deciding world state is CRITICAL.
- **No silent retcon** — enforced at ticket-write time: every ticket modifying existing behavior cites the change rationale in its Assumption Reassessment (`tickets/README.md`). This skill's contribution is catching drift introduced *between* reassessment and decomposition; durable architecture-changing decisions are gated by a foundational-doc amendment.
- **Phase acceptance gates (execution 02; foundation 12)** — any acceptance gate the decomposition would break (no-human run, deterministic replay, TUI playability, why-not explanations, missing-expected-property first-proof) is a CRITICAL Issue blocking decomposition until the user dispositions it.
