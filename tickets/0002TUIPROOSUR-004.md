# 0002TUIPROOSUR-004: Sealed debug capability + debug constructor sealing

**Status**: PENDING
**Priority**: HIGH
**Effort**: Large
**Engine Changes**: Yes — `tracewake-core` (new `DebugCapability`/`NonDiegeticDebugContext`; debug report/view constructors), `tracewake-tui` (debug renderers)
**Deps**: None

## Problem

Debug quarantine is enforced only by public booleans, marker strings, and runtime assertions: debug view structs expose public `debug_only: bool` and `non_diegetic_marker: String` (`crates/tracewake-core/src/view_models.rs:217-320`), debug report structs each carry public `debug_only` (`crates/tracewake-core/src/debug_reports.rs:27,45,52,59,66,…`), and debug renderers `assert!(report.debug_only)` at runtime (`crates/tracewake-tui/src/debug_panels.rs:13,37,61,…`) under `const DEBUG_MARKER` (`:10`). Any code can construct a falsely-marked debug view. Spec 0002 §4 TUI-AC-007 requires that debug reports/views which may contain privileged truth be constructible only with an unforgeable debug capability generated by core debug APIs; public bool/marker may remain *rendered output* but must not be the authority boundary.

## Architecture Check

1. A capability token with private fields makes false debug markers unconstructible by type, replacing a runtime `assert!` (which only fires if reached) with a compile-time guarantee — strictly stronger quarantine and the doctrine-preferred boundary (`INV-107`). It also lets the marker string remain as human-facing render output without being load-bearing.
2. No backwards-compatibility aliasing/shims: public `debug_only` authority fields are removed or demoted to derived render metadata, not kept as a parallel writable authority; no constructor bypass is retained.

## Verification Layers

1. INV-107 (debug quarantine) -> codebase grep-proof: privileged debug constructors require `DebugCapability`/`NonDiegeticDebugContext`; the capability has no public constructor outside core debug APIs.
2. INV-068 (non-diegetic) -> compile-fail / integration test: an attempt to build a privileged debug view/report outside debug APIs fails to compile or is structurally impossible.
3. INV-093 (no leakage) -> manual review: the embodied builder path (002) and notebook (010) are confirmed to never receive a capability.

## What to Change

### 1. Debug capability type

Add `crates/tracewake-core/src/debug_capability.rs` defining `DebugCapability` / `NonDiegeticDebugContext` with private fields, minted only by core debug/report APIs; declare the module in `lib.rs`.

### 2. Gate debug constructors

Make privileged debug-view/report constructors in `view_models.rs` and `debug_reports.rs` require the capability (or be private to the debug module). Demote `debug_only`/`non_diegetic_marker` to derived render metadata.

### 3. Renderers take debug types only

Update `debug_panels.rs` renderers to accept only debug view/report types (which now imply the capability), replacing runtime `assert!(debug_only)` authority with type-level guarantee.
